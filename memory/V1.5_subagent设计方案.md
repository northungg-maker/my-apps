# 观点类自媒体 V1.5 - Subagent 架构设计方案

## 一、核心架构理念

| 角色 | 职责 | 上下文特点 |
|------|------|-----------|
| **主 Session** | 调度器 + 用户交互 | 极轻，只做协调 |
| **Subagent-采集器** | 搜索 + 采集 | 独立上下文，有 Browser 能力 |
| **Subagent-写作** | 汇总 + 写作 | 独立上下文，干净环境 |

**核心优势：**
- 主 session 永远不崩（只做调度）
- 每个 subagent 独立运行，互不干扰
- 一个崩了不影响其他，可单独重跑

## 二、Subagent 切分方案

### 方案对比

| 方案 | Subagent 数量 | 优点 | 缺点 |
|------|--------------|------|------|
| A. 按来源类型 | 6个采集 + 1个写作 = 7个 | 并行高效 | 同时运行多个，管理复杂 |
| B. 按批次 | 每次新建（约14次） | 最细粒度 | 创建开销大 |
| **C. 混合方案** ✅ | **最多4个采集 + 1个写作 = 5个** | 平衡稳定与效率 | — |

## 三、执行流程

### 主 Session 逻辑（调度器）

```
1. 收到选题 → 确认是否执行 V1.5
           ↓
2. 创建 Subagent-官方采集
   - spawn(task="采集官方来源...", attachAs=专题文件夹)
   - 等待完成 → 收到结果 → 展示给老板 → 确认
           ↓
3. 创建 Subagent-专业采集
   - spawn(task="采集专业来源...")
   - 等待完成 → 展示 → 确认
           ↓
4. 创建 Subagent-评论采集
   - spawn(task="采集评论来源...")
   - 等待完成 → 展示 → 确认
           ↓
5. 创建 Subagent-写作
   - spawn(task="根据已采集素材写文章...")
   - 等待完成 → 展示成品
           ↓
6. 交付完成
```

### 串行执行 vs 并行执行

**推荐：串行执行（更稳定）**
- 官方 → 专业 → 评论 → 写作
- 每一步都等老板确认
- 便于调整和干预

**可选：并行执行（更快但复杂）**
- 官方 + 专业 + 评论 同时启动
- 全部完成后统一展示
- 适合老板不想频繁确认的场景

## 五、容错机制

### 如果 Subagent 崩了怎么办？

1. **主 session 检测到 subagent 异常**
   - 查看 subagent 日志
   - 判断是任务问题还是系统问题

2. **任务问题**（阈值太高、找不到来源）
   - 调整参数，重新 spawn 同类型 subagent
   - 或人工介入

3. **系统问题**（超时、资源不足）
   - 等待后重试
   - 或分割为更小的任务

### 文件持久化

每个 subagent 完成后，结果写入文件：
- 即使后续步骤失败，之前的工作不丢失
- 可以从任意步骤恢复

## 六、与 V1.4 的对比

| 项目 | V1.4 | V1.5 |
|------|------|------|
| 主 session 负载 | 中（每2网页需处理） | 极低（只调度） |
| 崩溃风险 | 有（上下文积累） | 几乎为零 |
| 恢复能力 | 从文件恢复 | 从任意步骤恢复 |
| 任务并发 | 串行但需主进程参与 | 可串行可并行 |
| 交互频率 | 每2网页一确认 | 每类来源一确认 |
| 成本 | 较低 | 较高（多个 subagent 运行） |

## 八、实现建议

### 主 Session Spawn 示例

```javascript
// 启动官方采集 subagent
sessions_spawn({
  runtime: "subagent",
  task: `你是【官方采集器】...`,
  cwd: 专题文件夹路径,
  attachAs: { mountPath: 专题文件夹路径 },
  mode: "run",
  runTimeout: 600  // 10分钟超时
})
```

### 监控 Subagent 状态

```javascript
// 主 session 可以定期查看
subagents({ action: "list" })

// 或等待 subagent 自动回报完成
```

## 总结

**V1.5 核心改变：**
- 主 session 从"执行者"变为"调度器"
- 每个任务环节由独立 subagent 完成
- 主 session 崩溃风险降至最低

**架构图一句话版：**
> 主 session 是项目经理，subagent 是执行团队，文件系统是共享知识库。
